Scott Armitage wrote:
> Actually the decorator pattern kind of subverts my issue with
> registering the callback with the core; mainly, that trusting the
> plugin author to take of that each and every time would be tedious and
> could introduce errors. If a decorator pattern took care of all that
> automatically, then you achieve the same thing I was thinking of --
> mainly, having a single code-base that handles the linking.

There is still some trust here to make sure the authors place the
decorator, but really who cares? We will maintain most of the plugins in
our tree so we won't really have a problem.

> Here's a nifty thought -- fresh off the presses in my mind, so I
> haven't worked through it yet -- but a callback function could specify
> which data/GUI controls it wants to have access to. When the plugin
> gets registered with the core (or whatever happens), an object is
> built that contains references to only those attributes. Then whenever
> that particular callback is called, it gets passed this object. Hmm,
> thinking about this a bit more, I think this might also have too much
> overhead. Something to come back to, definitely.

It sounds interesting, but I'm not sure either. More importantly, there
isn't a real "core" to Sonata so far. The base class is a mash of code
which we're slowly picking away at.

>  Attributes might be a bit cleaner, but would be more difficult to
> handle when missing (i.e. as opposed to plugin.get('key','default')).

Well, missing attributes can be defaulted with getattr(plugin, 'key',
'default') - so not too tricky.

> Sorry, I was starting to ramble. I tend not to think out what all I
> want to say in advance, and it just kind of comes as I type. I think I
> was trying to hammer home two disparate ideas here: first, that each
> plugin -- if contained within it's own module -- could be imported
> directly, from anywhere -- e.g. 'from SuperAwesomeGuysPluginPack
> import WickedCoolPlugin'. This is less of a concern, but it avoids
> having to bring in and parse the module manually if you could just do
> a simple import statement.

Ok, thanks for clarifying that. Makes sense to me and it's definitely an
advantage of leveraging the language. Of course, finding
SuperAwesomeGuysPluginPack is a bit tricky right?

> Good discussion so far. I think any patches from my end will be quite
> far away -- I'll be doing my work primarily locally and in Bazaar
> branches on Launchpad (http://www.launchpad.net/~scott-armitage
> <http://www.launchpad.net/%7Escott-armitage>), but I think there's a
> lot more to hash out first, especially if I'm going to try tackling a
> plugin revamp.

Great. I look forward to seeing something come out of this discussion.
There is definitely a lot more to figure out, but it doesn't mean you
can't hack away at something first and figure out where the next
roadblock is :-) At the least, I've started thinking more critically
about this.

> Thank you kindly for the input; I presume that Tuukka is subscribed to
> this list, so I'll not bother him(?) about it just yet, but certainly
> as I start getting a clearer picture of where I see this heading.

Yes he is.
