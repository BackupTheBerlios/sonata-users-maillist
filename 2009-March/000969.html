On Tue, Mar 31, 2009 at 12:04 PM, michael.venables <
michael.venables at ronin-group.org> wrote:

>  **I'm sort of confused about the way that the search function works in
> the Library tab. Just starting to type in the Current list filters results
> that contain the typed string -- and that makes sense. But it's not going to
> give individual results for "antonio" and "molina" if you type "antonio
> molina", only that string as a complete entity.
>
> In the Library tab, typing only jumps to the first hit from the first
> character, and moves on sequentially from there. So if I hit "s", I don't
> get everything with an "s", I jump to the first hit that BEGINS with "s",
> and the next character I type better match something in the esses, or the
> search is done. It would be really nice if search in the Library tab worked
> like it does in the Current. I think this is what Tomas is asking for, but
> I'm confused by Scott's answer. Is Sonata already supposed to behave like
> this?
>

You're confusing the built-in gtk find-as-you-type (if a gtk treeview has
focus and you start typing, it matches like you described above) with the
library searching that is performed when you type in the search entry below
the treeview. I assume Tomas was referring to the latter.

For the current tab, we override the built-in gtk find-as-you-type to
provide a more functional filtering capability. Indeed this could be done
for the library tab as well, but it isn't currently. I'd be worried about
the two capabilities conflicting and being confusing to the user, but I
agree it's already an imperfect state.

I recently had this conversation with someone (I thought it was you?). In
any case, that's where we stand.
