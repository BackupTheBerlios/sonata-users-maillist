> I think it's hard to show in screenshots so I will try to explain.
> Shoutcast streams usually have set only the song name tag. I don't want
> to have "Unknown - Unknown" in Current song line 2 when listening to
> them and so I have set the song line to {%A}{ - %B} to show nothing in
> such case. When I switch between such stream and a normal file (with
> Artist and Album name tags) the interface jumps similar to the (fixed)
> bug "interface jumping because of progress bar font".

Ahh, I see the issue now. It should be fixed, thanks.

> When I mark all at once and click to add them to the playlist it takes
> 10 seconds before the UI is responsible again.

I don't think there's much I can do about this.. I can make the UI
more responsive when doing this, but it's not going to shorten how
long the process takes. The beginning of each stream needs to be
retrieved to determine if it's a playlist or a stream, and then
depends wholly on how fast the stream's server sends data. The
checking and parsing work on Sonata's end is very quick.

> Now that I know about pls I have tested it too but it doesn't always

>
> http://www.idobi.com/radio/iradio.pls
> http://www.punkrockers-radio.de/listen.pls

I actually fixed this issue in svn right before I got your email ;) I
imagine it also fixes your issue with a lot of shoutcast streams not
working too.

> Some other thing: When I have a long playlist and doubleclick on a
> stream at the end of the list, the playlist jumps to the beginning.
> It would be better if it stayed where I had scrolled it.

Argh.. I had never noticed this before. Playing a stream actually
causes the mpd playlist to change/increment, which ends up refreshing
the current playlist, among other things. What a pain.. I have an ugly
workaround so that the playlist doesn't jump now, hopefully it doesn't
break anything.

For what it's worth, mpd streaming support is pretty poor/buggy..

> I can only speak for amazon.de. It has many covers that aren't

> works exactly like the main amazon.com site but some other players

I'll look into this at some point in the future.

> As mentioned above shoutcast streams have set only the song name tag.
> But this tag contains the artist AND the song name: "artist - song"
> Thus searching for lyrics should be possible but the lyrics tab shows
> "Artist or song title not set".

How is Sonata to know that both the artist and song are in the song
name tag? Is it supposed to assume that a '-' in the song means this?
There's obviously a huge number of situations where that doesn't work
(hyphenated words). Do other clients look for this sort of thing? I
hate to add specific code to work around problems with shoutcast
streams that don't properly set their tags.
