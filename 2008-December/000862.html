Hi all,

Manuel Naranjo wrote:
> Yesterday taling to tuukkah in the IRC we started talking about a
> plugins framework, and I told him I was going to try to implement a
> prototype. Well here it is, I present you my plugin framework. The
> plugin frameworks uses distutils so we leave distutils all the work
> related to dependencies, insertion points and all that. Actually the
> hole framework is based on this:
> http://lucumr.pocoo.org/blogarchive/setuptools-plugins

Thanks Manuel for this prototype! It already demonstrates the core 
parts: plugins can be installed separately, and Sonata doesn't need to 
know about them beforehand. Now to get further in the design, we can 
discuss what we ideally want from it.

I think these are essential:

  * Sonata can ship with a default set of plugins.
  * People can download and install new plugins.

These would be nice to have:

  * Sonata finds new plugins when they are put in ~/.sonata/plugins
  * Sonata chooses the newest version of multiple copies of a plugin.
  * People can configure and disable plugins.
  * Programmers can publish their plugins on the Sonata web site.
  * People can update and search for new plugins within Sonata.

What else? I must be missing some good ideas...
