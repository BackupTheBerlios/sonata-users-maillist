Well, it turns out to be marginally helpful - it helps some operations very
much (like Dan's add to playlist), but it's so poorly implemented that it
hardly helps many other operations. Removing a song from the playlist, for
example, will return the entire playlist if the removed song is at the top
of the playlist (while a removed song at the bottom of the playlist is sped
up). So I still use some manual model/treeview manipulations to speed up
these operations, which sucks.

Anyway, it's now implemented in svn, so adding to a large playlist should be
much faster. (Any testing would be great.) Other than that operation, I
think the only speed increase would be from using playlist manipulations
(like remove) through another mpd client, since my model/treeview
manipulation code isn't triggered.
