On Sat, Sep 20, 2008 at 5:13 PM, Dan McGee <dpmcgee at gmail.com> wrote:

> Just for another viewpoint, I run mpd with this size collection AND I
> run it remotely. I run mpd clients on 4 different machines (including
> the machine running mpd) to control it, so anything that makes
> performance drag would be no good for me.


You sound like a great test candidate then :) I just committed this
functionality in svn - if you could let me know how responsive it is, that'd
be helpful. And using the search field of 'everything' would be the most
taxing. It's quite responsive here, but then again I'm using localhost and
only have 2000 songs in my library. I quite like the behavior, I have to
say.

It uses many of the same techniques that the real-time playlist filtering
uses to help the performance (e.g. a gobject_idle delay between the end of a
keystroke and the beginning of a search, so that it doesn't have to search
on every keystroke if there are many fast keystrokes in a row).

Oh, and please report any bugs - my testing hasn't been too extensive and
I'm feeling a little rusty having not worked on the code much in recent
months.
