Hi again,

intrigeri wrote (13 May 2007 16:24:39 GMT) :
> But... IIRC, "modal" and "transient" are different things : the
> "transient" property is part of the ICCCM specification [1], and when
> the WM is informed a dialog is transient for an application, it can
> handle it in a special way. On the other hand, "modal" is a UI toolkit
> thing : a "modal" window has no particular ICCCM property, and the
> window manager has no standard means to know if it's modal or not, and
> thus it is neither supposed nor in a position to treat it in
> a special way.
>
> PyGTK reference manual is quite clear [2] on this topic : "Dialog
> windows should be set transient for the main application window they
> were spawned from. This allows window managers to keep the dialog on
> top of the main window, or center the dialog over the main window."

I digged a bit more into this, using the "xprop WM_TRANSIENT_FOR"
command, and it appears that the "about", "save playlist", and "edit
tags" windows are already set transient for the main window. Sorry,
I was confused to see, in sonata.py, a set_transient_for() for the
about dialog but none for the others.

After a bit more reading before complaining (sorry again), it appears
the problem is actually not specific to Sonata, but common to all GTK
applications : GTK sets the windows properties only after it has
mapped the window, i.e. told the window manager to start managing the
window. Well, my fault for using a ICCCM-compliant window-manager,
writen by someone who does not feel like implementing workarounds for
GTK apps.
