Scott Horowitz wrote (09 Apr 2007 00:20:22 GMT) :
> On 4/8/07, intrigeri <intrigeri at boum.org> wrote:
>
>     Sonata's integrated tag editor seems to pad every tag field with
>     spaces, in order, I guess, to save time on future tag edits. Great.
>
>     The issue (at least for me) is : this padding whitespace is displayed
>     as-is in the playlist and library views, which I find ugly, especially
>     when I edited one single track's tag in an disc, and thus only this
>     track has a bunch of spaces in the displayed information.
>
>     Would it be possible to hide such padding space in the library and
>     playlist views ?
>
>
> I'm not sure I understand exactly what you're saying. If you're finding tags
> displayed in Sonata with whitespace, it's because your music file's tags have
> whitespace ;) Are you asking for Sonata to trim all tag strings, i.e. remove
> leading and trailing spaces from all tags? Or something else?

My testcase is : a directory containing an album's tracks, properly
tagged except for the track number.

Before I do any tag editing :
  - the artist tag contains "The Doors"
  - the album tag contains "L.A. Woman"
  - one of the music files' title tag contains "The Changeling"
  - this album's listing in Sonata artist view is the following :
    http://intrigeri.boum.org/tmp/sonata_screenshots/before_tag_edit.png

Then I right click on the track called "The Changeling", and set the
correct track number, then save.

Then...
  - this track has disappeared from the album listing
  - a new artist has appeared in artist view, just after "The Doors",
    and it's called... "The Doors                     " (padded with
    spaces, yes)
  - this new artist has one album called "The Changeling           ",
    also padded with spaces, and this album has one track
  - this "new" album listing, with format '%A {- %B }{- %T }{- %S}{ (%L)}',
    is ugly, see
    http://intrigeri.boum.org/tmp/sonata_screenshots/after_tag_edit.png
  - And anyway, ncmpc shows the same ugly behaviour.

So... then, what do I really want ? I want my music player not to make
any difference between "The Doors" and "The Doors        ". I think this
should actually be fixed in mpd, to that any mpd client is fixed at
the same time. But :
  - I intend to maintain a Debian repository with up-to-date Sonata
    packages for Etch (= Debian stable), in order to distribute
    it easily to the various boxes I'm using.
  - I've never reported any feature request or bug to mpd, and thus
    I don't know how long it would take for it to be considered,
    implemented or rejected.
  - Even if this was implemented in mpd, I don't intend to maintain
    a Debian repository with a fixed version.

My conclusion is :
1. The elegant way to proceed would be to get this into mpd, and I'm
   going to do what can be done so that it happens.
2. In the meantime, a lot of people will go on using current mpd,
   simply because their distro does not provide the newer one ; then,
   I believe Sonata should do the trimming work itself, if 

What's your feeling about this ?
